MODULE 01
Web Security
Web Fundamentals for Hackers
HTTP, Headers, Cookies, Same-Origin Policy
1.1How HTTP Works (Attacker's View)

Every payment transaction travels over HTTP/HTTPS. Understanding the protocol at a low level is essential — attackers manipulate headers, methods, and payloads that developers often trust blindly.

ComponentSecurity Relevance
HTTP MethodsGET vs POST — GET params logged in server logs, never use for sensitive data
Request HeadersHost, Referer, Origin — forgeable by attackers for CSRF/SSRF
Response HeadersMissing security headers = open attack surface
Status CodesVerbose errors (500, 403) leak server internals
HTTP/2 PushCan be abused for cache poisoning attacks
Raw HTTP Request — what attackers see and modifyHTTP
POST /api/payment/process HTTP/1.1
Host: payback.app
Content-Type: application/json
Cookie: session=abc123; user_id=42
X-Forwarded-For: 127.0.0.1   <-- spoofable!
Origin: https://payback.app   <-- forgeable!

{"amount": 100, "to": "merchant_id_99"}
1.2Critical Security Headers

Missing headers are a free ticket for attackers. Set these on every PHP response:

✓ PHP — middleware/bootstrap
header("Content-Security-Policy: default-src 'self'; script-src 'self'");
header("X-Frame-Options: DENY");           // Prevent clickjacking
header("X-Content-Type-Options: nosniff"); // No MIME sniffing
header("Strict-Transport-Security: max-age=31536000; includeSubDomains");
header("Referrer-Policy: strict-origin-when-cross-origin");
header("Permissions-Policy: payment=(), camera=()");
1.3Cookies — The Session Currency

Payment sessions are prime targets. Cookie misconfiguration = account takeover.

✓ PHP — Secure cookie setup
session_set_cookie_params([
    'lifetime' => 3600,
    'path'     => '/',
    'domain'   => '.payback.app',
    'secure'   => true,      // HTTPS only
    'httponly' => true,      // JS cannot access — blocks XSS theft
    'samesite' => 'Strict'   // Blocks CSRF — use Lax for OAuth flows
]);
FlagMissing = vulnerable to…
SecureCredentials exposed over plain HTTP
HttpOnlyXSS can steal the session token via JS
SameSite=StrictCSRF attacks
SameSite=LaxGood for OAuth redirects — sent on top-level navigations
DomainSet to your domain only, never .com or overly broad
1.4Same-Origin Policy & CORS

Two URLs share an origin only if protocol, host, AND port all match. Misconfigured CORS in payment APIs is catastrophically common.

⚠ Dangerous — reflects any OriginPHP
$origin = $_SERVER["HTTP_ORIGIN"] ?? "";
header("Access-Control-Allow-Origin: " . $origin); // NEVER DO THIS
header("Access-Control-Allow-Credentials: true");
✓ Correct — whitelist specific originsPHP
$allowed = ["https://payback.app", "https://app.payback.app"];
if (in_array($origin, $allowed)) {
    header("Access-Control-Allow-Origin: " . $origin);
    header("Access-Control-Allow-Credentials: true");
}
Lab 01 — Inspect Real Traffic
  1. Install Burp Suite Community (free) — the core tool for all labs
  2. Set browser proxy to 127.0.0.1:8080
  3. Browse your local Payback dev environment, capture requests
  4. Find: What security headers are missing? Any cookies without HttpOnly?
  5. Challenge: Can you find an endpoint that reflects the Origin header back?
MODULE 02
Reconnaissance
Recon & Information Gathering
Subdomain Enum, Port Scanning, OSINT, Fingerprinting
2.1Why Recon Matters

Before any attack, an attacker spends 60–80% of time on reconnaissance. For a payment company, forgotten staging servers, exposed admin panels, and old API versions are goldmines. Know your attack surface before attackers do.

2.2Subdomain Enumeration
subfinder / amass / httpxbash
# Enumerate subdomains
subfinder -d payback.app -o subs.txt
amass enum -passive -d payback.app

# Check for dangling subdomains (subdomain takeover):
# If staging.payback.app points to an unclaimed cloud resource,
# an attacker can claim it and serve malicious content on your domain

# Probe which subdomains are alive:
cat subs.txt | httpx -title -status-code -tech-detect
2.3Port Scanning
nmapbash
nmap -sV -sC -p- --min-rate 5000 target.ip

# Dangerous exposed ports for payment companies:
# 3306 = MySQL direct (should NEVER be public)
# 6379 = Redis (no auth by default in older versions)
# 9200 = Elasticsearch (often no auth)
# 8080 = Admin panels, dev servers

# Check Docker containers aren't exposing ports publicly:
docker ps --format "table {{.Names}}\t{{.Ports}}"
2.4Technology Fingerprinting
whatweb / curlbash
whatweb https://payback.app
wappalyzer https://payback.app

# Check what your server leaks in response headers:
curl -I https://payback.app

# Bad: Server: Apache/2.4.41 (Ubuntu)  <-- version exposed
# Bad: X-Powered-By: PHP/8.1.2         <-- remove this!

# php.ini:  expose_php = Off
# Apache:   ServerTokens Prod
2.5OSINT for Payment Companies
  • GitHub dorking: search for payback + api_key OR password OR .env
  • Shodan.io — find exposed services tied to your IP ranges
  • Google dorks: site:payback.app filetype:env OR filetype:sql OR inurl:admin
  • LinkedIn — identify tech stack, employees for social engineering awareness
  • Wayback Machine — find old endpoints, forgotten pages still in production

MODULE 03
Injection
Cross-Site Scripting (XSS)
Reflected, Stored, DOM-Based, CSP Bypass
3.1XSS in Payment Context

XSS in a payment app is critical-severity. A stored XSS in transaction notes, invoice fields, or user profiles lets attackers steal session cookies, initiate fraudulent transfers, or modify the DOM to change payment amounts shown to users.

3.2Reflected XSS

User input is immediately reflected in the response without sanitization.

⚠ Vulnerable — search endpointPHP
$query = $_GET["q"];
echo "<h2>Results for: " . $query . "</h2>"; // NEVER DO THIS
✓ Fix — always encode outputPHP
echo "<h2>Results for: " . htmlspecialchars($query, ENT_QUOTES, "UTF-8") . "</h2>";
3.3Stored XSS — The Dangerous One

Stored XSS persists in the database and executes for every user who views the page. Payment description fields, invoice notes, and transaction memos are common injection points.

⚠ Vue — dangerous v-html patternVue
<template>
  <!-- DANGEROUS: never use v-html with user-controlled data -->
  <div v-html='transaction.description'></div>

  <!-- SAFE: Vue escapes by default with double-brace syntax -->
  <div>{{ transaction.description }}</div>
</template>
✓ If you must render HTML — use DOMPurifyJS
import DOMPurify from 'dompurify';
const clean = DOMPurify.sanitize(userInput);
3.4DOM-Based XSS

The payload never reaches the server — it executes entirely in the browser. Common in Vue/JS SPAs that read URL parameters or localStorage.

⚠ Vulnerable — unsafe DOM sinkJS
✓ Fix — use textContentTS
document.getElementById('greeting').textContent = 'Hello ' + name;

// TypeScript helper — enforce safe DOM manipulation:
function safeSetText(el: HTMLElement, text: string): void {
  el.textContent = text;
}
3.5Content Security Policy (CSP)
✓ Strong CSP headerHTTP
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{RANDOM_NONCE}';  // nonces > unsafe-inline
  style-src 'self' 'nonce-{RANDOM_NONCE}';
  img-src 'self' data: https://cdn.payback.app;
  connect-src 'self' https://api.payback.app;
  frame-ancestors 'none';  // Prevents clickjacking
  form-action 'self';
Common CSP Bypasses to Test
1. JSONP endpoints on whitelisted domains
2. Angular/Vue CDN with unsafe-eval
3. Open redirect on a whitelisted domain

XSS Test Payloads for Payment Fields

<script>alert(1)</script>
<img src=x onerror=alert(1)>
"><svg onload=alert(1)>
javascript:alert(1)
<iframe srcdoc="<script>alert(1)</script>">

Test these in: payment descriptions, invoice notes, user display names, search fields.

MODULE 04
Injection
SQL Injection
Manual SQLi, sqlmap, Blind Techniques, MySQL Hardening
4.1SQLi in Payment Databases

SQL injection in a payment system means attackers can read transaction records, extract credentials, modify payment amounts, or drop entire tables. Every query that touches user input is a potential entry point.

4.2Manual SQL Injection
⚠ Vulnerable — string concatenationPHP
$id = $_GET["transaction_id"];
$result = mysqli_query($conn, "SELECT * FROM transactions WHERE id = $id");

// Attack: transaction_id=1 OR 1=1--
// Result: Returns ALL transactions from ALL users

// UNION-based extraction:
// ?id=0 UNION SELECT username,password,3,4 FROM users--
✓ Fix — prepared statements alwaysPHP
$stmt = $pdo->prepare("SELECT * FROM transactions WHERE id = ?");
$stmt->execute([$id]);
$result = $stmt->fetchAll();
4.3Blind SQLi & sqlmap
Boolean-based & time-based blindbash
# Boolean-based blind:
?id=1 AND (SELECT SUBSTRING(password,1,1) FROM users WHERE id=1)='a'

# Time-based blind (infer from response time):
?id=1 AND IF(1=1, SLEEP(5), 0)  -- 5 second delay = condition is true

# sqlmap — automated exploitation:
sqlmap -u 'https://payback.app/api/transaction?id=1' --dbs
sqlmap -u 'https://payback.app/api/transaction?id=1' -D payback -T users --dump

# With Burp request file:
sqlmap -r request.txt --level=5 --risk=3
4.4MySQL Hardening
✓ Restricted DB user + audit loggingSQL
-- Restricted user for the application:
CREATE USER 'payback_app'@'localhost' IDENTIFIED BY 'strong_password';
GRANT SELECT, INSERT, UPDATE ON payback.* TO 'payback_app'@'localhost';
-- Do NOT grant: DELETE, DROP, ALTER, CREATE, FILE

-- Enable query logging for audit trail:
SET GLOBAL general_log = 'ON';
SET GLOBAL general_log_file = '/var/log/mysql/queries.log';

-- Disable LOAD DATA INFILE in my.cnf: local-infile=0
MODULE 05
Authentication
Authentication & Session Attacks
Brute Force, JWT, OAuth, Session Fixation
5.1Brute Force & Rate Limiting
✓ PHP — Rate limiting with Redis
function checkRateLimit(string $ip, string $username): bool {
    $key = 'login_attempts:' . md5($ip . $username);
    $attempts = (int) $redis->get($key);

    if ($attempts >= 5) {
        return false; // Blocked
    }

    $redis->incr($key);
    $redis->expire($key, 900); // 15 minute window
    return true;
}
5.2JWT Attacks
JWT attack vectorsbash
# Attack 1: algorithm confusion (none algorithm)
# Change header: {"alg": "none", "typ": "JWT"}
# Remove signature — some libraries accept this!

# Attack 2: alg:RS256 → HS256 confusion
# Sign with HS256 using the RS256 public key as secret

# Attack 3: weak secret brute force
hashcat -a 0 -m 16500 token.jwt wordlist.txt
✓ PHP — Secure JWT validationPHP
// ALWAYS specify allowed algorithms explicitly:
$decoded = JWT::decode($token, new Key($publicKey, 'RS256'));

// NEVER: JWT::decode($token, $key); // allows any algorithm
5.3OAuth Vulnerabilities
✓ PHP — CSRF-safe OAuth statePHP
// Generate and validate state parameter to prevent CSRF:
$state = bin2hex(random_bytes(16));
$_SESSION['oauth_state'] = $state;

// On callback:
if (!hash_equals($_SESSION['oauth_state'], $_GET['state'])) {
    throw new Exception('State mismatch — possible CSRF attack');
}
Open Redirect in redirect_uri
Bad: redirect_uri=https://payback.app/callback?next=https://evil.com
Validate redirect_uri against a strict whitelist of registered URIs.
5.4Session Security
✓ PHP — Session best practices
// 1. Regenerate session ID after login (prevent fixation):
session_start();
session_regenerate_id(true); // true = delete old session
$_SESSION['user_id'] = $user['id'];

// 2. Bind session to IP + User-Agent (detect hijacking):
if ($_SESSION['fingerprint'] !== hash('sha256', $_SERVER['REMOTE_ADDR'] . $_SERVER['HTTP_USER_AGENT'])) {
    session_destroy();
    redirect('/login');
}

// 3. Absolute session timeout for payment sessions:
if (time() - $_SESSION['created'] > 3600) {
    session_destroy(); // Force re-login after 1 hour
}
MODULE 06
Access Control
IDOR & Broken Access Control
BOLA, BFLA, Mass Assignment, Privilege Escalation
6.1IDOR in Payment Systems

IDOR (also called BOLA in APIs) is the #1 finding in fintech bug bounties. Attackers access other users' payment records by manipulating IDs.

⚠ Classic payment IDORPHP
$transaction_id = $_GET["id"];
$tx = $db->query("SELECT * FROM transactions WHERE id = $transaction_id");
// Attacker changes id=42 to id=43 — sees ALL transactions
✓ Fix — always enforce ownershipPHP
$user_id = $_SESSION["user_id"]; // Trust session, not user input
$stmt = $pdo->prepare("SELECT * FROM transactions WHERE id = ? AND user_id = ?");
$stmt->execute([$transaction_id, $user_id]);
$tx = $stmt->fetch();
if (!$tx) {
    http_response_code(404); // 404 not 403 — don't reveal ID exists
    exit;
}
6.2Mass Assignment Attacks
⚠ Blindly binding request body to modelPHP
$data = json_decode(file_get_contents('php://input'), true);
// Attacker sends: {"name": "Ahmed", "role": "admin", "balance": 99999}
$user->update($data); // Updates ALL fields including role and balance!
✓ Fix — whitelist allowed fieldsPHP / TS
$allowed = ['name', 'email', 'phone'];
$safe = array_intersect_key($data, array_flip($allowed));
$user->update($safe);

// TypeScript: use DTOs (Data Transfer Objects):
interface UpdateProfileDTO {
  name: string;
  email: string;
  // role, balance, id deliberately NOT included
}
6.3Privilege Escalation Patterns
  • Horizontal: user A accesses user B's data (same privilege level)
  • Vertical: regular user accesses admin functionality
  • Check: does removing admin role from JWT still allow admin actions?
  • Check: do admin API endpoints verify role, or just authentication?
✓ Centralized authorization middlewarePHP
function requirePermission(string $permission): void {
    $user = getCurrentUser();
    if (!$user->hasPermission($permission)) {
        audit_log('UNAUTHORIZED_ACCESS', $user->id, $permission);
        http_response_code(403);
        exit(json_encode(['error' => 'Forbidden']));
    }
}

requirePermission('transactions.refund');
requirePermission('admin.users.view');
MODULE 07
Server-Side
SSRF, File Upload & Path Traversal
SSRF, XXE, File Upload Attacks, Path Traversal
7.1SSRF — Server-Side Request Forgery

SSRF lets attackers make your server send requests to internal services — AWS metadata, internal admin panels, Redis, or microservices that trust internal IPs.

⚠ Vulnerable — webhook URL not validatedPHP
$url = $_POST['webhook_url'];
file_get_contents($url); // SSRF!
// http://169.254.169.254/latest/meta-data/ — AWS IAM keys
// http://localhost:6379 — Redis
// http://10.0.0.1:8080/admin — Internal admin panel
✓ Fix — validate and block private IPsPHP
function isSafeUrl(string $url): bool {
    $parsed = parse_url($url);
    if (!in_array($parsed['scheme'], ['http', 'https'])) return false;
    $ip = gethostbyname($parsed['host']);
    return !in_array(true, [
        str_starts_with($ip, '10.'),
        str_starts_with($ip, '192.168.'),
        str_starts_with($ip, '169.254.'),
        $ip === '127.0.0.1',
    ]);
}
7.2File Upload Attacks

Payment apps accept receipts, invoices, and identity documents. Unrestricted file upload = remote code execution on your PHP server.

✓ Secure file upload for payment receiptsPHP
function secureUpload(array $file): string {
    // 1. Validate file type by CONTENT (not extension or MIME header)
    $finfo = new finfo(FILEINFO_MIME_TYPE);
    $mimeType = $finfo->file($file['tmp_name']);
    $allowed = ['image/jpeg', 'image/png', 'application/pdf'];

    if (!in_array($mimeType, $allowed)) {
        throw new Exception('File type not allowed');
    }

    // 2. Rename file — never use original filename
    $ext = ['image/jpeg'=>'jpg', 'image/png'=>'png', 'application/pdf'=>'pdf'];
    $newName = bin2hex(random_bytes(16)) . '.' . $ext[$mimeType];

    // 3. Store OUTSIDE web root
    $dest = '/var/private/uploads/' . $newName; // Not /var/www/html/uploads/
    move_uploaded_file($file['tmp_name'], $dest);
    return $newName;
}
7.3Path Traversal
⚠ Vulnerable — file download endpointPHP
$filename = $_GET["file"];
readfile("/var/www/uploads/" . $filename);
// Attack: ?file=../../../../etc/passwd
✓ Fix — resolve and verify path prefixPHP
$filename = basename($_GET["file"]);
$realPath  = realpath("/var/www/uploads/" . $filename);
$uploadDir = realpath("/var/www/uploads/");

if (!str_starts_with($realPath, $uploadDir)) {
    http_response_code(403);
    exit;
}
readfile($realPath);

Testing with curlbash

8.2GraphQL Security
GraphQL attack vectorsGraphQL
# 1. Introspection — leaks entire schema (disable in production):
{ __schema { types { name fields { name } } } }

# 2. Batching attack — bypass rate limiting:
[{"query": "mutation { login(user:\"a\", pass:\"1\") }"},
 {"query": "mutation { login(user:\"a\", pass:\"2\") }"},
 ... 1000 login attempts in one HTTP request]

# 3. Deep query DoS:
{ users { friends { friends { friends { friends { id name } } } } } }
GraphQL Defenses
Disable introspection in production · Query depth limiting (max 5 levels) · Query complexity limiting · Persisted queries only
8.3API Fuzzing with Burp Suite
  • Use Burp Intruder to fuzz parameter values, IDs, and headers
  • Load SecLists payloads: /Discovery/Web-Content/api/objects.txt
  • Look for: 500 errors (code crashes), slower responses (timing), unexpected 200s
  • Fuzz transaction IDs: 1, 2, 3… 9999 — find gaps that reveal enumeration
  • Fuzz currency/amount fields: negative values, zero, very large numbers, strings
MODULE 09
Logic Flaws
Business Logic Vulnerabilities
Race Conditions, Price Manipulation, Workflow Bypass
9.1Race Conditions in Payments

Race conditions in payment processing are critical. If a withdrawal check and execution aren't atomic, attackers can double-spend by sending parallel requests. This is the most common critical bug in fintech bug bounty programs.

⚠ Vulnerable — double-spend race conditionPHP
function withdraw(int $userId, float $amount): bool {
    $balance = getBalance($userId);
    if ($balance >= $amount) {            // Check
        // --- Race window here ---
        deductBalance($userId, $amount);   // Use
        return true;
    }
    return false;
}
✓ Fix — SELECT FOR UPDATE (atomic row lock)
function withdraw(int $userId, float $amount): bool {
    $pdo->beginTransaction();
    try {
        $stmt = $pdo->prepare('SELECT balance FROM accounts WHERE id = ? FOR UPDATE');
        $stmt->execute([$userId]);
        $balance = $stmt->fetchColumn();

        if ($balance < $amount) { $pdo->rollBack(); return false; }

        $pdo->prepare('UPDATE accounts SET balance = balance - ? WHERE id = ?')
             ->execute([$amount, $userId]);
        $pdo->commit();
        return true;
    } catch (Exception $e) { $pdo->rollBack(); throw $e; }
}
9.2Price Manipulation
⚠ Price from POST body — attacker sends price=0.01PHP
$price = $_POST['price'];   // Attacker sends price=0.01
$qty   = $_POST['quantity'];
chargeUser($price * $qty);  // Pays $0.01 instead of $100
✓ Fix — fetch price from server-side onlyPHP
$product_id = (int) $_POST['product_id'];
$qty        = (int) $_POST['quantity'];
$product    = $db->fetch('SELECT price FROM products WHERE id = ?', [$product_id]);
chargeUser($product['price'] * $qty); // Price from DB, not user
9.3Workflow Bypass
  • Can users reach /order/confirmation without completing payment?
  • Can users jump from step 1 to step 5 in a multi-step KYC flow?
  • Can the same coupon be applied multiple times in parallel requests?
  • Can a refund be initiated on an already-refunded transaction?
✓ Enforce workflow state server-sidePHP
$allowed_transitions = [
    'cart'     => ['checkout'],
    'checkout' => ['payment'],
    'payment'  => ['confirmation'],
];
$current_step   = $_SESSION['order_step'] ?? 'cart';
$requested_step = $_POST['next_step'];
if (!in_array($requested_step, $allowed_transitions[$current_step] ?? [])) {
    die('Invalid workflow transition');
}

10.3Sample Report Template
Title
IDOR in GET /api/v1/transactions/{id} exposes full payment history
Severity
High (CVSS 7.5)
Summary
The transaction detail endpoint does not verify that the authenticated user owns the requested transaction. By iterating the numeric ID parameter, any authenticated user can view full payment details of all other users.
Steps to Reproduce
1. Log in as user A — obtain JWT token
2. Make a payment — note your transaction ID (e.g. 1042)
3. Send: GET /api/v1/transactions/1041 with user A's token
4. Response contains user B's payment details
Impact
Full financial transaction history of all users exposed. PCI DSS violation. Regulatory reporting may be required.
Remediation
Add ownership check: WHERE transaction_id = ? AND user_id = ?. Return 404 (not 403) to avoid confirming ID existence.
